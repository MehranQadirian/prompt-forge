# Fix Expo Dev Site `npm ci` Build Failure

## Problem

EAS build runs `npm ci --include=dev` and fails because the `package-lock.json` is out of sync with what the Expo build environment expects. The lock file lists many packages only as peer dependencies (e.g., `react-dom`) without resolving concrete versions, and is missing `@react-native/babel-preset` entirely.

## Root Cause

The lock file was generated locally and doesn't fully resolve all transitive/peer dependencies that EAS build needs. `npm ci` is strict — every package in the lock file must have a concrete resolved entry, and peer dependencies that EAS explicitly requires (like `react-dom` for web support) aren't being resolved.

## Fix (2 steps)

### Step 1: Delete the stale lock file
```bash
rm package-lock.json
```

### Step 2: Regenerate a fresh lock file
```bash
npm install
```

This forces npm to resolve ALL dependencies from scratch, including:
- `react-dom` (peer dep needed for web/EAS builds)
- `@react-native/babel-preset` and its `@babel/plugin-transform-*` children
- All other transitive dependencies with proper version resolution

### Step 3: Verify
```bash
npm ci --include=dev
```

This should now succeed because the fresh lock file will have all entries properly resolved.

## Files Modified
- `package-lock.json` — regenerated (deleted then recreated by `npm install`)

## Why This Works
- `npm install` resolves the full dependency tree including peer deps that npm previously skipped
- The fresh lock file will be in exact sync with `package.json`
- EAS build's `npm ci --include=dev` will find all expected packages present
